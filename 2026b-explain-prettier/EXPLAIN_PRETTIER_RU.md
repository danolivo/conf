# EXPLAIN Prettier или упрощение планов запросов

Эта история началось с подаренной книги: читая книгу Jimmy Angelakos [«PostgreSQL Mistakes and How to Avoid Them»](https://www.amazon.com/PostgreSQL-Mistakes-How-Avoid-Them/dp/163343687X), я осознал один напрягающий меня факт — команда EXPLAIN выдаёт слишком много информации. И примеры, которые авторы обычно приводят рассматривая тот или иной аспект систем баз данных, усложняют разбор задачи и рассеивают внимание.
Так и родилась идея пост фильтра для эксплейнов - чтобы сделать их более читабельными и проблемно-ориентированными.

## Перегруженность деталями

Каждый, кто работал с PostgreSQL, знает команду `EXPLAIN`, а точнее `EXPLAIN ANALYZE`. Обычно она испольуется в расследовании проблем производительности запроса или демонстрации техники оптимизации. Но есть одна беда: её вывод наполнен множеством специфической информации. Например, при анализе EXPLAINa достаточно редко требуется параметр width. Некоторые поля, например `cost` занимают много места, системно-зависимы и в принципе не нужны - однако если мы хотим увидеть величину `planned rows`, то нам приводится делать эксплейн с опцией `COSTS OFF` и `cost` попадает в него безальтернативно.
И что, разве это большая проблема - изучать эксплейн с яуть большим количеством информации? Ха! Давайте посмотрим на типичный план запроса в моих расследованиях - вот например два плана одного запроса, один условно [плохой](https://explain.tensor.ru/archive/explain/894c945f2c9e050f314860f139009d5f:0:2026-02-09), а второй - [хороший](https://explain.tensor.ru/archive/explain/0677212a1b01c7988be25d160f1a0c03:0:2026-02-09).
Поиск проблем в таком большом плане занимает время и каждая лишняя деталь мешает разглядеть проблемное решение в коде. Да, с приходом AI агентов я могу просто попросить Claude сравнить пару планов, выдать мне различия и проанализировать, что не так. Однако это не всегда работает, либо деталей чересчур много, либо нужна автоматизация на большом пространстве запросов - в общем, проблема остаётся.

## Стабильность регрессионных тестов

Другой аспект - это стандартизация тестов. EXPLAIN меняется между версиями Postgres и если ваше расширение поддерживает 4-5 крайних версий, то и тесты должны проходить на каждой из них. Таким образом, нужно фильтровать вывод эксплейнов, чтобы гарантировать стабильность прогона в разных программно-аппаратных условиях. Пример:

```
Seq Scan on users (cost=0.00..1000.00 rows=100)
  Filter: status = 'active'
  Buffers: shared hit=50
  Heap Fetches: 5
```

А на другой машине или версии PostgreSQL выходит чуть иначе:

```
Seq Scan on users (cost=0.00..995.50 rows=99)
  Filter: status = 'active'
  Buffers: shared hit=48
  Heap Fetches: 3
```

CI/CD конвейер падает. Не потому что план изменился, а потому что различия платформ сломали точное совпадение. Знакомо?

А вот ещё более простой случай. Возьмём тривиальный запрос с сортировкой:

```
Sort (actual time=0.027..0.028 rows=1 loops=1)
  Sort Key: id
  Sort Method: quicksort  Memory: 25kB
  ->  Seq Scan on users (actual time=0.009..0.010 rows=1 loops=1)
```

Значение `Memory: 25kB` зависит от платформы. На другой машине с другим аллокатором или выравниванием памяти вы получите `Memory: 26kB` или `Memory: 24kB`. И ведь в `EXPLAIN ANALYZE` нет никакой опции, чтобы подавить вывод этого поля — оно печатается всегда, когда узел Sort выполняется в памяти. То же касается узлов Hash, где появляются `Memory Usage`, `Buckets`, `Batches`. Вы не можете попросить PostgreSQL «покажи мне план без деталей памяти» — такой настройки просто не существует.

С `pretty_explain_analyze()` эта проблема исчезает:

```sql
-- Раньше: хрупкие тесты с точным совпадением
SELECT * FROM my_test_function()
EXCEPT
SELECT * FROM expected_output;

-- Теперь: стабильно на разных версиях и оборудовании
SELECT pretty_explain_analyze('SELECT * FROM users WHERE status = ''active''')
EXCEPT
SELECT pretty_explain_text($$
  Seq Scan on users (rows=100)
    Filter: status = 'active'
$$);
```

Больше никаких ложных сбоев. Тесты проходят последовательно, независимо от версии PostgreSQL и оборудования.

## Планы запросов для статей и документации


Ну и наконец - литература. Демонстрируя читателю, например, преимущества BitmapScan, не обязательно загружать его сознание лишней информацией, как `loops` или `width`. Представьте: вы пишете статью о новой фиче оптимизатора запросов PostgreSQL и хотите показать планы до и после:

```
До оптимизации:
Hash Join (cost=1000..5000 rows=1000)
  Buffers: shared hit=500 read=100
  -> Seq Scan on customers (cost=0.00..500.00)
       Memory Usage: 42kB

После оптимизации:
Nested Loop (cost=100..500 rows=1000)
  Buffers: shared hit=50
  -> Index Scan on customers_idx (cost=0.00..100.00)
       Memory Usage: 8kB
```

Размеры памяти, буферы, оценки затрат — всё это загромождает картину и к тому же будет другим на машине читателя. А ведь важно здесь одно: как изменилась структура плана. А должно быть чисто, компактно. Читатель должен видеть суть оптимизации без лишнего шума.




## Что происходит под капотом

По умолчанию `pretty_explain_analyze()` и `pretty_explain_text()` нормализуют вывод EXPLAIN. Скрывается всё, что зависит от платформы: распределения памяти (`42kB` превращается в `NN`), количество рабочих потоков (`Workers Planned: 8` становится `Workers Planned: N`), шум с плавающей точкой (`rows=100.00` упрощается до `rows=100`), значения времени округляются до целых. Буферы, выделение памяти хеша и прочие детали реализации удаляются.

При этом сохраняется всё важное: структура плана — узлы, объединения, сканирования; фильтры и условия; оценки количества строк. Оценки затрат тоже можно включить при необходимости.

## Управление выводом

Все флаги по умолчанию выставлены на `false` — для максимальной стабильности. Но если нужно больше деталей, это легко настроить:

```sql
-- Максимальная фильтрация (по умолчанию) — только структура плана
SELECT pretty_explain_analyze('SELECT ...');

-- Показать всё (для отладки или анализа производительности)
SELECT pretty_explain_analyze('SELECT ...',
  platform_dependent => true,
  show_cost => true,
  show_details => true
);

-- Показать затраты, но скрыть остальные детали
SELECT pretty_explain_analyze('SELECT ...',
  show_cost => true
);
```

## Быстрый старт

Для регрессионных тестов:
```sql
SELECT pretty_explain_analyze('SELECT * FROM your_query');
```

Для копирования/вставки из psql:
```sql
SELECT pretty_explain_text($$
  [Вставьте вывод EXPLAIN сюда]
$$);
```

Для сравнения планов:
```sql
WITH plan1 AS (SELECT pretty_explain_analyze('SELECT ...')),
plan2 AS (SELECT pretty_explain_text('$$...$$ '))
SELECT * FROM plan1 EXCEPT SELECT * FROM plan2;
```

## Вместо заключения

EXPLAIN Prettier решает одну конкретную проблему: планы запросов слишком шумны для того, чтобы их можно было нормально сравнивать и тестировать на разных версиях и платформах. Он автоматически убирает неважные детали реализации, сохраняя структуру плана — и в результате тесты перестают ломаться по пустякам, документация не устаревает при обновлении версий, планы из production и тестового окружения честно сравниваются, а примеры в статьях не зависят от конкретной версии PostgreSQL.

Вместо того чтобы вычищать вывод EXPLAIN вручную, можно наконец сосредоточиться на том, ради чего всё затевалось — на понимании и оптимизации запросов.
